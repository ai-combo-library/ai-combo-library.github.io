Firecrawl + Haystack

低成本验证“网页资料采集到可检索知识库”的核心链路，适合先跑小样本，再决定是否接入 语义向量服务、向量库或完整问答 UI。

文档与知识 · 实测通过 · 有使用限制 · 适合特定人群有实测证据含限制说明
FirecrawlHaystack
适合谁想把产品文档、帮助中心、网页资料先变成可检索文本的小团队；正在评估 RAG/知识库原型，但不想一开始部署重系统的人；需要保留采集端和检索端可替换空间的开发者
不适合谁需要现成聊天 UI、权限协作和后台运营的团队；需要抓登录态、复杂反爬或高并发网页采集的场景；已经明确需要生产级向量库和多模型评估的项目
最大的限制本轮只覆盖单个公开小网页和 BM25 内存检索；未验证复杂站点、登录页、批量抓取、语义向量服务 成本、向量库持久化和 LLM 回答质量。

我们怎么测

使用 Firecrawl API 抓取一个公开网页并取得 Markdown，把该文本写入 Haystack 内存文档库，建立 BM25 索引后用预设关键词查询。

实际结果

Firecrawl 返回了有效 Markdown，Haystack 成功索引并把预期文档排在检索结果中，3 项检查通过；结论只适用于小样本 BM25 原型。

实际工作流怎么走

  1. 1. Firecrawl：获取原始资料，并把内容交给下一步。
  2. 2. Haystack：接收上一步结果，并完成这一环节的处理。

继续补成可长期使用方案要做什么

  1. 固定测试样本和最终产物
  2. 用同样样本重新跑一遍完整流程
  3. 保留失败原因
  4. 保存完整实测记录
  5. 定期重新确认是否仍然可用
